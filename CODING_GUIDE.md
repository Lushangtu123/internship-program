## 4. AI 的强制执行协议

### 4.1 开始前：先理解，再实现

1. 阅读项目说明、架构文档、相关模块、现有测试与最近变更；使用快速检索定位相似实现，避免重复造轮子。
2. 明确列出假设、风险、依赖、范围外事项和可验证的成功标准。存在会显著影响方案的歧义时，先提问，不得静默猜测。
3. 对新功能、组件、行为变更或创意实现，先完成需求/设计探索：提供 2–3 个可行方案、说明取舍、给出推荐方案，并在获得确认后实施。简单任务可使用简短设计，但不能跳过设计确认。
4. 实施前完成信心检查。只有总体信心 **≥90%** 才可开始编码；应检查：
   - 是否已有重复实现；
   - 是否符合项目架构与既有约定；
   - 需要时是否查阅了官方文档；
   - 复杂或陌生方案是否有可信参考实现；
   - 缺陷类任务是否已定位根因。
5. 若需要隔离开发，先检查当前是否已在受管 worktree；仅在必要且获同意时创建新的 worktree。开始改动前运行相关基线检查；若基线失败，先报告并等待决定。

### 4.2 实施中：最小、可追溯、可测试

1. 仅修改完成本需求所必需的文件和代码；所有改动必须能直接追溯至本文件中的一项需求或验收标准。
2. 优先使用项目现有的技术栈、组件、命名和编码风格；未经同意不得引入新框架、依赖、大规模重构或未要求的“顺手优化”。
3. 选择最简单、最小的可维护方案。不要为单次使用抽象通用层，不要预埋未要求的配置或功能。
4. 对功能、修复、重构和行为变更采用 TDD：先写一个表达期望行为的最小测试，确认其因功能缺失而失败；再写最少实现使其通过；最后在保持全绿的前提下重构。配置或一次性原型等例外必须先说明并获得同意。
5. UI 工作应遵循对应框架和可访问性规范，并在真实浏览器/目标设备尺寸下验证关键路径。只有在任务涉及视觉判断时才生成原型、截图或视觉对比。
6. 每个阶段报告：已完成内容、修改位置、执行的验证、结果、剩余风险和下一步；不要把“看起来可行”当作完成。

### 4.3 发现缺陷时：先根因，后修复

1. 完整阅读报错、日志、调用栈与失败测试，并给出稳定复现步骤。
2. 检查最近改动、环境差异、配置和依赖；跨服务/跨层链路应在边界采集证据，追踪错误数据的来源。
3. 在代码库中寻找相同场景的正常实现，比较差异并形成单一、可证伪的根因假设。
4. 用最小实验验证假设；确认根因后，为问题补充失败测试，再实施单一修复并运行回归验证。
5. 若连续三次修复均未解决问题，停止叠加补丁，重新审视架构并与需求方讨论。

### 4.4 完成前：验证、审查、交付

1. 逐条映射并验证所有验收标准，至少覆盖正常、边界与错误路径。
2. 运行受影响测试、全量相关测试、静态检查、类型检查、构建及格式化检查；若任何检查无法运行，说明原因和潜在影响。
3. 对重要功能、复杂修复或合并前变更执行独立代码审查，优先处理严重和重要问题；审查范围应包含需求符合性、回归风险、测试完整性、安全性和可维护性。
4. 最终交付必须包含：改动摘要、关键文件、验收结果、执行命令与输出摘要、已知限制/未做项，以及后续建议（仅在确有必要时）。

## 5. 已安装 Coding Skills：按场景选择

AI 应根据任务实际触发下列能力，不应为了“使用 skill”而增加不必要步骤。带“必用”的规则优先执行。

| 场景 | 优先使用的 skills | 期望产出 |
| --- | --- | --- |
| 新功能、组件、交互或行为变化 | `brainstorming`（必用）、`Confidence Check`（实施前）、`test-driven-development`（必用）、`karpathy-guidelines` | 已确认的设计、≥90% 信心检查、先失败后通过的测试 |
| Bug、构建/测试失败、性能异常 | `systematic-debugging`（必用）、`diagnose`、`Confidence Check`、`test-driven-development` | 复现、证据、根因假设、回归测试和单一修复 |
| 既有代码改动或多步计划 | `using-git-worktrees`、`create-plan`、`planning-with-files`、`codebase-migrate` | 隔离与基线状态、实施计划、阶段性验证 |
| 代码质量与合并前检查 | `requesting-code-review`、`review`、`writing-guidelines` | 独立审查结论、已处理问题、验证证据 |
| Web 前端 | `frontend-dev` / `frontend-design`、`web-design-guidelines`、`webapp-testing`、`playwright` | 可访问、响应式 UI；浏览器自动化验证 |
| React / Next.js | `vercel-react-best-practices`、`vercel-composition-patterns`、`vercel-optimize`、`vercel-react-view-transitions` | 与 React 生态相符、性能受控的实现 |
| Vue / Nuxt | `vue`、`vue-best-practices`、`vue-router-best-practices`、`vue-testing-best-practices`、`vueuse-functions`、`pinia`、`nuxt` | Composition API + TypeScript 优先、可测试的状态与路由 |
| 移动端 | `ios-application-dev`、`android-native-dev`、`react-native-dev`、`vercel-react-native-skills`、`flutter-dev` | 平台规范一致、真机/模拟器验证计划 |
| 后端、全栈、Agent/LLM 系统 | `fullstack-dev`、`project-development`、`mcp-builder`、`tool-design`、`evaluation` / `advanced-evaluation` | 清晰边界、接口契约、评估与回归策略 |
| 测试工具与构建工具 | `vitest`、`vite`、`pnpm`、`tdd` | 与项目工具链一致的测试和构建配置 |
| 部署与生产运维 | `deploy-to-vercel`、`deploy-pipeline`、`vercel-cli-with-tokens`、`gh-fix-ci`、`sentry-triage`、`datadog-logs` | 可回滚的部署步骤、CI/监控验证 |
| GitHub/PR 工作流 | `gh-address-comments`、`pr-review-ci-fix`、`changelog-generator` | 已处理评审、CI 状态、面向用户的变更说明 |
| 资料、API 或 OpenAI 产品 | `openai-docs`、`anysearch` / `smart-search`（仅在需要检索时） | 以官方一手资料为主的已验证结论 |
